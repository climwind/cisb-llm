# CISB Specification

## Vulnerability Description

**Source**
69102311a5

**Description**
GCC optimizer at -O2 may replace a conditional store (inside a branch guarded by a lock) with an unconditional speculative store. This violates the developer's assumption that the store is only executed under lock protection, leading to data races and memory corruption.

**Evidence**
With allow-store-data-races=1 (default), a conditional store in the kernel ring buffer code is transformed into an unconditional store. The condition was only expected to be true when logbuf_lock was held, but the speculative store occurred regardless, corrupting log_first_seq and triggering segfaults. Using allow-store-data-races=0 disables the optimization and the code behaves correctly.

**Requirement**
GCC version 4.3.4 (or other versions with allow-store-data-races optimization enabled by default), optimization level -O2, concurrent execution (e.g., NMI handler vs. regular kernel context), architecture that supports speculative store hoisting (e.g., x86).

**Mitigation**
Add compiler flag `-fno-allow-store-data-races` or `--param=allow-store-data-races=0` to disable speculative store optimization. Alternatively, upgrade to newer GCC that is more conservative about this transformation, or insert compiler barriers (`asm volatile` or `OPTIMIZER_HIDE_VAR`) around the conditional store to prevent hoisting.
